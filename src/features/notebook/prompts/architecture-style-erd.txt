## Style: ERD — Entity-Relationship Diagram

Draw an **entity-relationship diagram** for the data model. The audience is data engineers, DBAs, and domain modellers. **Design the target data model** — apply third normal form unless the sources explicitly justify denormalisation (analytics warehouse, read model).

Set `kind: "erd"`.

Use these element kinds:
- `entity` — exactly one per logical table / aggregate root. Each entity carries:
  - `label`: PascalCase singular noun (`Customer`, `Account`, `OnboardingApplication`, `TransactionLeg`).
  - `fields`: the column list. Each field has:
    - `name`: snake_case (`customer_id`, `created_at`, `iban`).
    - `type`: concrete SQL type (`uuid`, `varchar(64)`, `numeric(18,2)`, `timestamptz`, `jsonb`, `text`).
    - `notation`: a short tag — `PK` (primary key), `FK` (foreign key), `NN` (not null), `UQ` (unique). Combine if needed: `PK NN`.
  - `stereotype`: optional — `<<aggregate root>>`, `<<lookup>>`, `<<event store>>`, `<<view>>`.
  - `inferred`: `true` if the entity isn't explicitly named in the sources.

Relations are foreign keys / associations:
- `kind`: `"fk-1-1"`, `"fk-1-n"`, or `"fk-m-n"`. Use `"fk-m-n"` only when there's a real join table; usually you'll create the join entity explicitly and use two `"fk-1-n"` instead.
- `label`: the **business relationship**, not the FK column ("places" between Customer and Order, "owns" between Customer and Account, "consists of" between Order and OrderLine).
- `source` / `target`: on `fk-1-n`, the "1" side is `source` and the "n" side is `target`.

Design discipline:
- **5–15 entities.** Anything bigger is multiple ERDs (one per bounded context).
- Default to **third normal form**. Surrogate `uuid` primary keys + a natural-unique constraint (`UQ`) on business keys (IBAN, ISIN, email).
- Always include audit columns when the domain is regulated: `created_at`, `updated_at`, `created_by`, `version` (optimistic locking). Mark these `inferred: true` if the source didn't call them out.
- Soft-delete columns (`deleted_at`) only if the domain requires retention (legal hold, GDPR right-to-erasure tracking) — and call it out in the description.
- For monetary fields, use `numeric(18,2)` (or higher precision for FX / crypto) — never `float`. For dates, prefer `timestamptz`.
- Stereotype event-sourcing tables (`<<event store>>`) and read-model projections (`<<projection>>`) when the architecture pattern calls for them.

Do **not** use `person`, `system`, `external-system`, `container`, `component`, `deployment-node`, or `actor` in this style. Use only `entity`.

A good ERD answers: "What are the business invariants?", "Where do FK violations get caught?", "Which columns are queryable / indexed?". Encode that intent in your choice of keys, uniqueness, and not-null flags.
