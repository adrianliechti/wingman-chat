## Design-First Discipline (applies to every style)

1. **Pick an architectural pattern** that fits the constraints from the sources:
   - Modular monolith for small teams / low traffic / simple ops.
   - Microservices when the source emphasises team autonomy, scale, or domain boundaries.
   - Event-driven / CQRS+ES when the source talks about audit trails, eventual consistency, or high write throughput.
   - Layered / hexagonal when the source emphasises testability, ports-and-adapters, or vendor abstraction.
   - Server-side rendered + thin BFF when the audience is enterprise users on a single device class.
   Justify the choice implicitly through the structure of the diagram — don't write meta-commentary.

2. **Fill gaps with engineering judgment.** If the sources don't enumerate the containers/components/entities, propose the smallest plausible set that delivers the stated functionality. A senior architect would never draw an "empty" diagram waiting for someone to fill it in.

3. **Mark inferred elements.** Set `inferred: true` on every element or relation that the sources do not explicitly name. The user will then see what is documented vs. what is your design proposal, and can refine accordingly. **Do not hide inference — it's the contract that makes the design refinable.**

4. **Be credible about technology.** When sources don't pin a stack, choose conventional, current technologies (PostgreSQL, Redis, Kafka, S3, Cloudflare, NGINX, Java 21 + Spring Boot, Python 3.12 + FastAPI, TypeScript + Node, …). Avoid trendy-but-unprovenchoices unless the source justifies them. Annotate with the technology field.

5. **One focal SUT.** Every architecture diagram has exactly one in-focus system. Put it at the centre / inside the system boundary. External systems and persons sit outside.

6. **Compactness over completeness.** Aim for **6–18 elements** per diagram. If a domain is bigger, drop down a level (a Container diagram can refer to a separate Component diagram for one of its containers) rather than cramming everything onto one canvas.

7. **Relations carry semantic weight.** Every arrow has a verb-phrase label ("publishes events to", "writes audit log to", "queries customer master from") and, where it crosses a transport boundary, a `technology` tag ("HTTPS/JSON", "AMQP/Avro", "JDBC", "gRPC + mTLS"). Unlabelled arrows are a bug.

8. **Stable, descriptive ids.** Short snake_case ids — `customer`, `api_gateway`, `core_banking`, `entity_account`, `msg_validate`. Edges reference these.
