## Style: Data Contracts (ODCS)

Render the catalog as a set of **per-dataset data contracts** aligned with the Open Data Contract Standard (ODCS, Bitol Foundation, LF AI & Data). A data contract is a commitment between a data producer and its consumers: *what* the dataset is, *for what use*, with *which quality bars*, under *which SLAs*.

Set `kind: "contracts"`.

**Populate heavily**: `contracts` and the underlying `datasets` they reference. **Populate lightly**: `glossary` (only the terms that show up in contract `purpose` statements), `lineageNodes` / `lineageEdges` (skip — separate concern).

ODCS mapping cheat-sheet:
- ODCS `info` block → `Dataset` itself (id, name, title, description, owner, steward).
- ODCS `models` block → `Dataset.fields` (column-level schema).
- ODCS `purpose` → `DataContract.purpose`.
- ODCS `quality` → `DataContract.qualityRules` (plain-English testable statements).
- ODCS `terms.usage` / `terms.limitations` / `terms.noticePeriod` → fold into `DataContract.terms[]` as named term/commitment pairs.
- ODCS `servers` / `support` → not modelled separately; carry as `Dataset.system` / `Dataset.contact`.

Design discipline for contracts:
- **One contract per critical dataset.** Cover 4–10 datasets — the ones a downstream consumer would actually depend on. Don't write contracts for staging tables or temporary marts.
- **`purpose` is the audience-facing sentence.** "This dataset publishes confirmed trades for downstream risk and finance consumers. It is the system of record for trade attributes between booking and settlement." Concrete, scoped, useful.
- **`qualityRules` must be testable.** Good: "No NULL values in `trade_id`", "`notional_amount` ≥ 0", "`as_of_date` is the previous business day for batch runs", "Row count vs. source-of-record within 0.01%". Bad: "Data is accurate", "High quality".
- **`terms` are SLAs, not aspirations.** Pick canonical term names:
  - **Availability** — "Available 24×7 except Sunday 02:00–04:00 UTC maintenance window."
  - **Freshness** — "Latest record available by T+1 06:00 UTC for batch; sub-second for streaming."
  - **Retention** — "Retained 7 years per SOX requirements; 10 years for trades per MiFID II."
  - **Security** — "Row-level access controlled via `risk_role` group; column-level masking on `pii_email`, `pii_phone`."
  - **Change notice** — "Breaking schema changes announced ≥ 30 days in advance via #data-platform channel."
  - **Support** — "Escalation via #risk-data; production incident SEV-2 SLA 4h."
- **Versioning.** Every contract has a `version` (semver). Default `v1.0.0` if you're proposing the first version. Mark `inferred: true` since you are designing the contract.
- **Glossary touch-points.** When a contract's `purpose` uses defined terms ("counterparty", "exposure", "EOD"), add a minimal `GlossaryTerm` entry so cross-references work.

Most contracts will be `inferred: true` — that is expected. The user will refine each one with their organisation's actual SLA / quality bars.
