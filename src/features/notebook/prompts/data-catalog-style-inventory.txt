## Style: Inventory (DCAT 3)

Render the catalog as a **dataset inventory** aligned with W3C DCAT 3. This is the view a data-governance committee or BCBS 239 reviewer expects to see: every dataset listed once, with the metadata an auditor would want for due-diligence.

Set `kind: "inventory"`.

**Populate heavily**: `datasets` array. **Populate lightly**: `glossary` (only terms directly referenced by datasets), `lineageNodes` / `lineageEdges` (a few high-level arrows showing system-of-record → derived store relationships), `contracts` (only for the 2–3 most critical datasets).

DCAT mapping cheat-sheet:
- `Dataset.name` → `dct:identifier` (fully-qualified, stable).
- `Dataset.title` → `dct:title` (human-friendly).
- `Dataset.description` → `dct:description` (one paragraph; what is this dataset for?).
- `Dataset.owner` → `dct:publisher` (an organisation function: "Credit Risk", "Trading Operations").
- `Dataset.steward` → operational contact (technical team).
- `Dataset.domain` → `dcat:theme` (data product / bounded context: "Credit", "Markets", "Customer").
- `Dataset.system` → the storage technology hosting the dataset.
- `Dataset.location` → `dcat:downloadURL` / `dcat:accessURL` analogue (FQN, URI, topic).
- `Dataset.refreshCadence` → `dcat:accrualPeriodicity` analogue.
- `Dataset.sensitivity` → `dct:accessRights` ("public" / "internal" / "confidential" / "restricted").
- `Dataset.regulatoryTags` → `dcat:keyword`.

Design discipline for inventory:
- **Cover the full lifecycle** — operational source-of-record + DWH staging + analytics marts + downstream regulatory extracts. Don't only list the "interesting" tables.
- **Group by domain** — every dataset has a `domain` so the renderer can group them. Use the canonical domain names from the sources, or infer finance-standard names: Customer / Party, Product, Trade, Position, Risk, Finance / GL, Reference Data, Reporting.
- **Be honest about systems** — name the actual technology. "Snowflake" not "data warehouse". "Apache Kafka" not "messaging system". "MS SQL Server" if that's what the bank actually runs.
- **Field lists** — for the top 5–8 datasets, populate `fields` with PK / FK / classification / nullable. Skip field lists for reference-data lookups; mention `(see source schema)` in the description instead.
- **Sensitivity defaults** — anything containing customer identifiers or contact details is at least `confidential`; PII columns get `classification: "PII"` and the dataset gets `regulatoryTags: ["GDPR-PII"]`. Trading P&L and position data is `restricted`. Static reference data (currencies, countries) is `internal` or `public`.

Do **not** invent ontology references for glossary terms in this style — keep glossary minimal (just label + definition + dataset cross-refs).
