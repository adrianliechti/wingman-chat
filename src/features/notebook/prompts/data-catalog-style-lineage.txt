## Style: Lineage (OpenLineage)

Render the catalog as a **data lineage diagram** aligned with the OpenLineage spec (LF AI & Data). This is the view a BCBS 239 reviewer wants when asking "show me how this number was produced": every transformation from system-of-record to downstream consumer.

Set `kind: "lineage"`.

**Populate heavily**: `lineageNodes` and `lineageEdges`, plus the underlying `datasets` referenced by `LineageNode.datasetId`. **Populate lightly**: `glossary` (skip), `contracts` (skip — they don't render in this view).

OpenLineage mapping cheat-sheet:
- A **Dataset** in OpenLineage maps to a `LineageNode` with `kind: "dataset"` and a `datasetId` pointing to the underlying `Dataset` row.
- A **Job** in OpenLineage maps to a `LineageNode` with `kind: "job"` and a `technology` tag naming the tooling (dbt, Airflow, Spark, Glue, Fivetran, Talend, Informatica PowerCenter, Matillion).
- An **External system / namespace** maps to a `LineageNode` with `kind: "external"` (e.g. a vendor API, a regulator submission endpoint, an FX rate provider).
- An edge from a dataset to a job represents an **input**. An edge from a job to a dataset represents an **output**. Stick to that direction — `dataset → job → dataset`.

Design discipline for lineage:
- **Cover the in-scope flow end to end.** A good lineage shows: source-of-record → ingestion job → raw layer → transformation job → curated layer → consumption job → consumer (mart, report, model, API).
- **Always model the job, never connect dataset-to-dataset directly.** Even a 1-line CTAS counts as a job (name it, give it a technology). This is the OpenLineage invariant.
- **Name jobs as verb-noun.** `Ingest trades from Kafka`, `Normalise customer master`, `Compute EOD exposure`, `Publish FR-Y14 extract`.
- **Tag every job with a real `technology`** — pick one: `dbt`, `Airflow`, `Spark`, `Flink`, `Glue`, `Fivetran`, `Talend`, `Informatica`, `Matillion`, `Trino`, `bash`. Don't write "ETL" or "Python script" — pick the tool that fits the stack implied by the sources.
- **Use `LineageEdge.kind`** to colour the arrows: `ingest` (source → first job), `transform` (job → curated dataset), `publish` (final dataset → external), `replicate` (DR / cross-region copy).
- **6–25 lineage nodes.** Beyond that, the graph becomes unreadable — split into multiple lineage outputs by domain.
- **External producers and consumers are nodes too.** A regulator gateway (`MAS, FCA, FINRA, BaFin`) is an `external` node. So is a market-data vendor (`Bloomberg B-Pipe`, `Refinitiv RIC`).
- **Inferred jobs are explicit.** If the source says "data flows from A to B" but doesn't name the job, you must still create one — set `inferred: true` on it with a plausible name + technology.

Datasets referenced by lineage nodes need at least a stub entry in `datasets` (id, name, title, system). Fields and contracts are optional in this style.
