## Style: C4 — Container

Draw a **C4 Level 2 (Container)** diagram. Each container is a separately deployable / runnable unit: a web app, an API service, a single-page app, a database, a queue, a worker. **Not** a Docker container specifically — Simon Brown's C4 sense.

Set `kind: "c4-container"`.

Use these element kinds:
- `person` — primary external personas (1–3 max — keep the focus on the system internals).
- `system` — the SUT itself. Place all `container` elements inside it via `parent: <sut-id>`, **or** group them under a `system-boundary` group.
- `external-system` — collaborators outside the SUT boundary.
- `container` — the deployable units inside the SUT. **Every container has a `technology` tag** (e.g. "TypeScript / React SPA", "Java 21 / Spring Boot 3.3", "PostgreSQL 15", "Apache Kafka", "AWS S3", "NGINX").

Use **one** group of `kind: "system-boundary"` for the SUT; reference it from each container's `parent` field, or assign the `system` element to it.

Design discipline:
- 4–10 containers per diagram. If you exceed this, drop down to multiple Container diagrams scoped per bounded context.
- Default to a sensible separation: **edge (NGINX / API gateway) → application services → asynchronous workers → datastores → external integrations**.
- Pick a pattern explicitly: layered, hexagonal, microservices, event-driven, modular monolith. The shape of the diagram reflects the choice.
- Datastores are containers too. Stereotype them `<<datastore>>` and tag with the engine ("PostgreSQL 15", "DynamoDB", "Redis 7"). Queues / topics: `<<queue>>` / `<<topic>>`.
- For every container the source did not name explicitly, set `inferred: true` — including obvious infrastructure like caches, reverse proxies, message brokers.

Relation labels should describe the **action** and **direction**: "reads customer master from", "publishes onboarding-completed events to", "renders pages to". Tag relations that cross a process boundary with `technology`:
- "HTTPS/JSON"
- "AMQP" / "Kafka topic"
- "JDBC" / "psycopg"
- "gRPC + mTLS"
- "S3 PutObject"

Do **not** use `component` here (that's the next level down) or `actor` / `entity` (different styles).
